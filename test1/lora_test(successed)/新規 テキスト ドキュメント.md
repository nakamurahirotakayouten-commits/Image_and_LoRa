## Lora explain part

Loraモジュールは常にアンテナからいろんな電波を拾っていて，監視しているんだけれど，
電波に乗って送られている情報は下のようになっている．
```
1バイト目 (ADDRH)：送信先アドレス上位8bit
2バイト目 (ADDRL)：送信先アドレス下位8bit
3バイト目 (CH)：送信先チャンネル
4バイト目以降：実際のデータ
```
ここでLora評価モジュールは本体のflashメモリに保存されている自分のデバイスアドレスとチャンネルを参照して，
この電波は自分に向けて飛ばされたものかを判定している．

結果，自分にあててのものであれば，esp32にuartでその情報を送る．

esp32の方ではイベント駆動で，常にその情報を受け入れる準備ができていて，それがさっきの while(1)のループ のこと．

## memo2 part

あと，なぜか初めの方に通信がうまくいかなかったのは，
- 透明通信にしていなかったからなのか，
- ブレッドボードのuartで使った部分に電荷がたまっていて，変な感じだった？
- ↑は違って，単純に接触不良だった可能性が高いかも．．


、**3時間悩んでいた本当の問題**はこれだ！！

> **「LoRaRecvTask がループしていない」と思い込んでいたこと。**

実際には、

* `LoRaRecvTask()` はちゃんと動いていた。
* `ReceiveFrame()` の `while(1)` もちゃんと回っていた。
* しかし

```cpp
while (SerialLoRa.available()) {
```

に処理が一度も入らなかった。=受信バッファに届いていなかった．

つまり、**タスクは止まっていたのではなく、受信データ待ちをしていただけ**でした。

デバッグを進めていくうちに

```cpp
SerialMon.printf("now searching...");
```

をいろいろな場所へ入れた結果、

```
LoRaRecvTask
    ↓
ReceiveFrame
        ↓
while(1)
        ↓
SerialLoRa.available() == 0
```

という状態であることが分かりました。

> **「なぜ E220 が UART に受信データを出してこないのか」**

そこで，**透明通信に切り替えた**．**ブレッドボードのいろんな配線のところにLEDを並列につないで電流が流れているか・入力に応じて変化しているか確かめた**．

そしたら 突然 受信しだした．
